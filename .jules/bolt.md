## 2024-05-24 - O(N^2) Performance Bottleneck in Line Counting via String Slicing
**Learning:** Found a critical performance bottleneck in `src/scanner/analyzer.py` and reversing tools where `code[:match.start()].count("\n")` was used to find the line number of regex matches. Since `code` could be very large (entire files), slicing from the beginning of the string for every single match created an O(N^2) bottleneck, creating massive memory allocations and string copies. A simple benchmark on a mock large file showed slicing took 0.15s while incremental took 0.0016s (98x speedup).
**Action:** Replaced it with an incremental `count("\n", last_idx, match.start())` approach where `last_idx` carries forward. When using regex match indices (`match.start()`) to calculate line numbers, always use incremental string offset counting rather than slicing `str[:idx]` to prevent O(N^2) DoS.
